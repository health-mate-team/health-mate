# 오우너 디자인 룰북

> 이 문서는 모든 디자이너, 개발자, 에이전트(AI 도구 포함)가 오우너 앱을 만들 때 반드시 지켜야 할 규칙입니다. 이 룰을 어기면 디자인 일관성이 깨지고, 결국 브랜드가 약해집니다.

## 🎯 디자인 철학 5원칙

오우너의 모든 디자인 결정은 다음 5원칙을 거쳐야 합니다.

### 원칙 1: 캐릭터가 먼저, 데이터는 나중

> "건강 데이터는 모아의 상태를 통해 표현된다."

- 큰 숫자(체중, 칼로리)를 화면 중앙에 띄우지 않는다
- 차트보다 캐릭터의 표정과 자세로 상태를 먼저 보여준다
- "수분 50%"보다 "모아가 목말라요"가 우선

### 원칙 2: 따뜻함은 라운드와 코랄로

> "오우너는 친구이지, 도구가 아니다."

- 모서리는 항상 둥글게 (`radius.md` 이상)
- 핵심 액션은 코랄 컬러 (#FF7B5C)
- 흑백/회색 톤만으로 구성된 화면 금지

### 원칙 3: 부담을 주지 않는다

> "사용자가 앱을 안 켜는 것도 괜찮다."

- 강제, 협박, 죄책감 유발 메시지 금지 ("3일 동안 안 들어왔어요!")
- 알림은 격려와 초대로만 ("모아가 기다려요")
- 실패 화면 없음. "잠시 쉬어가요"로 표현

### 원칙 4: 한 화면에 한 가지 일

> "스크롤 길게 만들지 말고, 화면을 더 만들어라."

- 한 화면의 핵심 액션은 1개
- 보조 액션은 최대 2개 (탭바 제외)
- 정보 밀도가 높으면 화면을 분리한다

### 원칙 5: 모션은 살아있음을 표현

> "정적인 화면은 죽은 화면이다."

- 모아는 항상 살아있어야 한다 (호흡, 깜빡임)
- 상태 변화는 반드시 모션으로 표현
- 단, 거슬리지 않게 — 미세한 모션이 정답

## 🎨 컬러 사용 룰

### ✅ 반드시 지킬 것

**1. Semantic 토큰 우선**
```dart
// ✅ 좋은 예
Container(color: OwnerColors.bgPrimary)
Container(color: OwnerColors.actionPrimary)

// ❌ 나쁜 예
Container(color: OwnerColors.coral500)  // Semantic으로 명확한 의미가 안 보임
Container(color: Color(0xFFFF7B5C))     // 토큰 우회
```

**2. 한 화면에 코랄 한 곳만**
- 코랄은 가장 강한 시각 요소
- 한 화면에 코랄 영역이 2개 이상이면 → 우선순위 재검토
- 예외: 시그니처 하트(모아) + CTA 버튼은 함께 가능

**3. 컬러 = 의미**
- 코랄 = 액션, 브랜드, 에너지
- 베이지 = 따뜻함, 캐릭터, 배경
- 코코아 = 텍스트, 안정감
- 민트 = 운동, 성공
- 스카이 = 수분
- 라벤더 = 휴식
- 오렌지 = 별, 보상 (수의 시그니처)

새로운 의미가 생기면 → 토큰부터 추가, 그 다음 사용

### ❌ 절대 하지 말 것

- 그라디언트 사용 금지 (브랜드 정체성 모호해짐)
- 세 가지 이상의 컬러를 한 컴포넌트에 사용 금지
- `Colors.red`, `Colors.blue` 같은 Material 기본 컬러 직접 사용 금지
- Hex 코드 직접 입력 금지 (`Color(0xFF...)`)

## ✏️ 타이포그래피 룰

### 폰트는 Pretendard 하나만

- 다른 폰트 사용 금지
- 굵기는 400, 500, 600, 700만 사용 (300, 800 등 금지)
- 9개 정의된 스타일만 사용 (`display`, `h1~h3`, `body`, `bodySm`, `caption`, `button`, `overline`)

### 텍스트 위계 룰

```
h1  → 화면당 1개만 (페이지 타이틀)
h2  → 섹션 헤더, 한 화면에 여러 개 가능
h3  → 카드/모달 타이틀
body → 본문, 가장 많이 사용
caption → 보조 정보, 메타 데이터
overline → 카테고리 레이블, 상단에 미리 알리는 용도
```

### 텍스트 길이 룰

- **버튼**: 한국어 6자 이내, 영어 2단어 이내
- **h1**: 한 줄로 끝나야 함 (스마트폰 화면 기준)
- **h2/h3**: 두 줄까지 허용
- **body**: 4줄 넘으면 영역을 분리하거나 "더 보기" 처리

### 한국어 특화

- letter-spacing은 음수값 (-0.5~-1px) — 한국어가 너무 벌어져 보임
- 줄바꿈은 의미 단위로 (`<br>` 이용해 어색한 끊김 방지)
- 숫자는 영문 폰트로 별도 처리하지 않음 (Pretendard에 한글+영문+숫자 모두 포함)

## 📐 레이아웃 룰

### 4의 배수만

```dart
// ✅ 좋은 예
EdgeInsets.all(16)   // base
EdgeInsets.all(24)   // xl
SizedBox(height: 12) // md

// ❌ 나쁜 예
EdgeInsets.all(15)   // 4의 배수 아님
SizedBox(height: 18) // 4의 배수 아님
```

### 화면 좌우 여백은 16px 고정

- 모든 화면의 좌우 패딩 = `OwnerSpacing.base` (16px)
- 예외: 풀블리드 카드(인스타 스토리식 풀스크린 모멘트) 시 0
- 예외: 가로 스크롤 리스트 시 첫 카드만 16px, 그 다음은 8px

### 카드 내부 패딩 = 20px (lg)

- 정보 카드: `OwnerSpacing.lg` (20px)
- 작은 칩/태그: `xs` × `sm` (4×8)
- 빅 액션 카드: `xl` (24)

### 섹션 간격 = 24px (xl)

- 섹션과 섹션 사이 = `OwnerSpacing.xl` (24px)
- 카드와 카드 사이 = `OwnerSpacing.md` (12px) 또는 `base` (16px)

## 🎭 캐릭터 사용 룰

### 모아와 수의 등장 비율 = 7:3

- 모아는 메인이다. 어디든 등장할 수 있다.
- 수는 특별한 순간에만 — 보상, 미션, 챌린지, 큰 성취
- 두 캐릭터가 함께 나오는 경우 → 모아가 더 크게 (1.4배 비율 유지)

### 캐릭터 크기 가이드

| 용도 | 사이즈 |
|------|--------|
| 인라인 (텍스트 옆) | 24~32px |
| 헤더 아바타 | 44px |
| 카드 일러스트 | 60~80px |
| 화면 메인 | 140~220px |
| 풀스크린 모멘트 | 240~300px |

### 캐릭터 호흡 모션

- 모아는 항상 살아있어야 함
- 정지 상태에서도 미세한 호흡 모션 (scale 1.0 ↔ 1.02, 2.4초 사이클)
- 사용자 액션 후 0.3초 내에 반응 모션 (눈 깜빡, 살짝 끄덕)

### 절대 하지 말 것

- 모아를 회색조로 표현 금지 (죽은 것처럼 보임)
- 모아를 화면 모서리에 잘리게 배치 금지
- 모아 위에 반투명 오버레이 올리기 금지 (slept/disabled 상태도 별도 표정 사용)

## 🔘 컴포넌트 룰

### 버튼 위계는 3단계만

```
1차: ElevatedButton  - 코랄 배경 + 흰 텍스트   (한 화면에 1개)
2차: OutlinedButton - 흰 배경 + 코랄 테두리   (보조 액션)
3차: TextButton     - 텍스트만                (취소, 무시)
```

### 카드 종류는 3가지만

```
1. Surface Card      - 흰 배경 + 0.5px 테두리 (기본)
2. Elevated Card     - 코랄50 배경            (강조)
3. Hero Card         - 코랄500 배경 + 흰 텍스트 (CTA)
```

새로운 카드 스타일이 필요하면 → 디자인 시스템 회의에서 논의 후 추가

### 입력 필드

- 높이 48px 고정
- 라운드 8px (`radius.md`)
- 포커스 시 코랄 테두리
- placeholder는 `cocoa.200`

### 아이콘은 Lucide 또는 Material Outlined만

- 다른 아이콘 라이브러리 혼용 금지
- 한 화면에 두 라이브러리 혼용 금지
- 커스텀 아이콘이 필요하면 → SVG로 별도 자산 등록

## ⚡ 모션 룰

### Duration 매핑

```
사용자 즉각 피드백  → motion.fast (120ms)
화면 전환           → motion.base (240ms)
콘텐츠 등장         → motion.slow (400ms)
캐릭터 액션         → motion.character (800ms)
진화 연출           → motion.evolution (2400ms)
```

### Curve 매핑

```
일반 UI                → curve.standard (cubic 0.4, 0, 0.2, 1)
캐릭터, 보상, 알림     → curve.bouncy (cubic 0.34, 1.56, 0.64, 1)
페이드, 슬라이드 페이드 → curve.gentle (cubic 0.25, 0.1, 0.25, 1)
```

### 절대 하지 말 것

- 1초 이상 사용자를 기다리게 하지 말 것 (캐릭터 모션 제외)
- 동시에 4개 이상 모션 트리거 금지 (시각 피로)
- 흔들림, 진동 모션 금지 (스트레스 유발)

## 🔄 새로운 디자인 추가 프로세스

새로운 컴포넌트나 패턴이 필요할 때:

```
1. 기존 토큰/컴포넌트로 만들 수 있는가?
   YES → 그걸로 만든다. 끝.
   NO  → 2번으로

2. 새 토큰이 필요한가? 아니면 새 컴포넌트만 필요한가?
   토큰  → 토큰 파일에 먼저 추가, 팀 리뷰
   컴포넌트 → 기존 토큰을 조합한 위젯 생성

3. 새 토큰 추가 시 체크리스트
   [ ] Core 또는 Semantic 어디에 속하는가?
   [ ] 다크모드 대응 가능한가?
   [ ] 기존 토큰과 의미가 겹치지 않는가?
   [ ] 이름이 명확한가? (역할 기반)
   [ ] README에 사용 가이드 추가했는가?

4. 팀 리뷰 후 머지
```

## 🤖 AI 에이전트를 위한 가이드

Claude Code 같은 AI 도구로 화면을 생성할 때, 다음 순서로 작업하세요:

```
1. /screens/{screen_name}.json 명세 읽기
2. 명세의 components 배열 순회
3. 각 component마다 정의된 token만 사용
4. 명세에 없는 색상/사이즈 사용 금지
5. 의문이 들면 → 디자인 룰북 참조 → 그래도 안 되면 디자인 시스템 회의로
```

화면 명세 JSON 구조:
```json
{
  "screen_id": "home_character",
  "tokens_used": ["bgPrimary", "actionPrimary", ...],
  "components": [...],
  "interactions": [...]
}
```

이 형식이 유지되어야 자동화 도구가 일관성을 보장할 수 있습니다.
